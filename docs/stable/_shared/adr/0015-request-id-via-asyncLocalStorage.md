# Request-ID via `AsyncLocalStorage`

Every HTTP request is tagged with a `requestId` â€” a UUID generated by the first middleware in the chain, or honored from an incoming `x-request-id` header if the client provides one. The id is also echoed back in the response's `x-request-id` header so clients can correlate. The id is stored in an `AsyncLocalStorage` (from `node:async_hooks`) so any code running within the request's continuation can read it without it being threaded through function signatures. The logger reads from this store and includes `requestId` on every log line emitted during the request.

## Why `AsyncLocalStorage` instead of explicit threading

Alternative: pass the Hono context (`c`) or the `requestId` explicitly down through every service and repository call. Rejected because:

- Pollutes every signature with an HTTP-context parameter even when the layer is HTTP-agnostic (service + repository should not know they're being called from an HTTP request â€” they could be called from a CLI or scheduled job too).
- Defeats the cleanliness of [ADR-0008](./0008-backend-layer-model.md)'s layer model (services + repos are format-agnostic).
- Forces ad-hoc threading at every new method.

`AsyncLocalStorage` solves this transparently: middleware wraps the request handler in `requestContext.run({ requestId }, next)`, and any async code spawned during that continuation can call `requestContext.getStore()?.requestId` without knowing about Hono. The cost is ~1% perf overhead in worst-case benchmarks â€” acceptable.

## Why honor `x-request-id` from the client

Allows distributed correlation. If the frontend (or another upstream service) attaches a correlation id, the backend reuses it. If absent, we generate. The response always echoes the id back so the client can find it in their network tab and in any error toast they show.

## Scope

- Middleware: `apps/backend/src/utils/request-context.ts` exports the `AsyncLocalStorage`. A `requestIdMiddleware` (mounted as the first middleware in `createApp`) generates / honors the id and wraps `next` in `requestContext.run(...)`.
- Logger: reads `requestContext.getStore()?.requestId` when formatting each log line.
- Outside a request (CLI / migration / boot): the store is `undefined`; logger omits the field.
