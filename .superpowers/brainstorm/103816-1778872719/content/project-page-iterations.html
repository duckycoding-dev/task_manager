Project page — 3 ways to slice tasks vs reminders

Same underlying data; different mental model. Open Qs in terminal after picking.

X · Entity primary · View mode secondary (recommended)

Top toggle = Tasks ↔ Reminders (what you're looking at). Inside Tasks, secondary Board / List toggle (how you view tasks). Inside Reminders, just a list — board has no meaning for reminders (no status enum). Cleanest mental model. No awkward "All" combination.

Y · View primary · Entity as filter

Top toggle = Board / List. Filter chip selects entity: Tasks (default) / Reminders / All. "All" only makes sense in List view → if user is on Board and switches filter to Reminders or All, view auto-switches to List. Reactive but coupling makes the model heavier to explain.

Quick-add inside a board column · the inline-expanding row

Click "+ Add task" at the bottom of a column → the button transforms into an editable row in place. Type title, Enter saves, Esc cancels. Status + projectId are inferred from the column you clicked + this project.

Why inline instead of modal: stays in context (you see where the new card lands), keyboard-only flow (click button → type → ⏎ → repeat), no focus-trap dialog overhead.
Defaults applied silently: status = column · projectId = this project · priority='medium', isRecurring=false, recurringInterval='none', dueDate=null (column quick-add doesn't presume a date — user adds via detail panel).
After ⏎: new card appears at column bottom with brief pulse + "click to edit ✎" hint. Input stays focused for the next entry. Esc / Done collapses the input back to a "+ Add task" button.
