Interactions summary

  • Inline create row at top — swatch (click opens color popover with 12 presets + custom hex) + name input + Enter to create. Default color = hash-derived from name (same trick as projects). After create, row stays focused and clears for rapid creation.
  • Click name in a row → inline editable (same pencil→✓/✗ pattern as task title? — or click directly since this is a more lightweight context). Pick one. Recommended: click-to-edit directly here (lighter than pencil for short text in a table row).
  • Click swatch in a row → opens the color popover; immediately commits on selection.
  • Filter → action → navigates to All-tasks with the label preselected as a filter chip (`/tasks?labelId=<id>`).
  • Delete → confirmation modal showing impact count. Soft-delete (planned, like tasks/reminders).
  • Sort → mono toggle "Sort: usage ↓ / name ↑ / recent ↓". Persisted via URL.
  • Filter by name → client-side string match across the loaded list.

Backend changes for v1

🟢 Add labels.color as NOT NULL — today it's nullable. UX wants every label to have a color; if missing on create, fallback to hash-derived default. Migration backfills existing rows with the hash.
🟢 Soft delete on labels — same pattern as tasks/reminders. deletedAt: timestamp nullable. List endpoints filter deletedAt IS NULL.
🟢 GET /labels with per-label task count — either compute via JOIN on task_labels + tasks (deleted-filtered), or expose a separate GET /labels/:id/usage. Decide at backend phase. (Mockup shows counts; without this, the column is empty in v1.)

Mobile: same single-column table; row actions collapse into a ⋯ menu when the viewport narrows. Filter chip "Filter →" still goes to All-tasks page.
